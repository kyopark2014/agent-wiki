# Karpathy의 Second Brain, 실제 비즈니스에 적용하면?

> 원문: [LinkedIn Pulse - What Karpathy's Second Brain Looks Like Inside a Real Business](https://www.linkedin.com/pulse/what-karpathys-second-brain-looks-like-inside-real-business-taylor-rr2dc/)
> 저자: Adam Taylor (TCD Group)

---

## 🔑 핵심 배경

Karpathy의 LLM 기반 개인 위키 아이디어에서 영감을 받아, **TCD Group**(40년 역사의 광업 서비스 회사)이 이스터 연휴 동안 실제 비즈니스에 이를 구현했다.

> **핵심 인사이트**: *"합성(Synthesis)이 지능을 만든다 — 단순한 검색(Retrieval)이 아니라."*

- RAG 파이프라인 없음
- 벡터 데이터베이스 없음
- 약 40만 단어 규모에서는 불필요
- **모든 것이 로컬 실행** — 데이터가 외부로 나가지 않음 (민감한 상업 계약 보호)

---

## 🏗️ TCD Group의 4계층 아키텍처

Karpathy의 개인용 3단계를 비즈니스 규모에 맞게 **4개 레이어**로 확장:

| 레이어 | 내용 |
|--------|------|
| **Layer 1 — Raw Files** | PDF, 스프레드시트, 계약서, 안전 보고서 등 원본 문서 추출 및 분류 |
| **Layer 2 — Domain Summaries** | 프로젝트별 도메인 요약 (CONTRACT, SAFETY, ESTIMATE 등 구조화된 Markdown) |
| **Layer 3 — Per-Project Wiki** | 프로젝트당 하나의 종합 위키 페이지 (모든 도메인 요약 통합) |
| **Layer 4 — Cross-Project Intelligence** | 클라이언트/공급업체/상업 인텔리전스를 수년·수백 건의 계약에 걸쳐 집계 |

### 📊 규모
- **1,800개 이상의 프로젝트**
- **500,000개 이상의 파일**
- 각 프로젝트, 클라이언트, 주요 인물, 공급업체마다 위키 페이지 보유
- IMS(통합 관리 시스템)도 위키에 연결 → **지식 그래프** 형성

## ✅ 실제로 가능해진 5가지

### 1. 🧮 신입 견적사가 20년 베테랑처럼 견적 작성
- 새 프로젝트 입찰 시, 위키가 유사 프로젝트의 계약 금액, 실제 BOQ 단가, 승인된 변경 사항, 장비 성능, 우수 하청업체 정보를 즉시 제공
- 시니어 견적사 퇴직 시 사라지던 **기관 지식(Institutional Knowledge)이 이제 축적됨**

### 2. 🦺 실제로 사고를 예방하는 안전 인텔리전스
- 수천 건의 위험 보고서와 사고 기록이 위키에 교차 참조
- 예: "교대 첫날, 첫 1시간에 수동 취급 부상 급증" → 해당 날 사전 브리핑 강화
- 무사고 프로젝트 vs 반복 사고 프로젝트의 SWMS 비교 분석 가능

### 3. 📋 PM이 클라이언트 미팅을 며칠이 아닌 몇 분 만에 준비
- 기존: 3개 시스템에서 데이터 수집에 반나절 소요
- 현재: 프로젝트 사이트 페이지에 인력 현황, 안전 성과, 선행 지표, 위험 종결률 등이 **항상 최신 상태로 통합**

### 4. 📄 문서 컴플라이언스 자동화
- IMS 내 1,110개 통제 문서 중 22개가 검토 기한 초과 → **위키가 즉시 플래그**
- 특정 문서가 12개 활성 프로젝트에서 참조됨 → 문서 개정 시 **어떤 안전 계획을 업데이트해야 하는지 자동 파악**

### 5. 🏭 실제 성과 데이터 기반 공급업체 선정
- 784개 공급업체 페이지 구축
- 시간 관리, 품질, 안전, 환경 성과, PM 추천 여부로 **점수화**
- 유사 작업 경험 공급업체를 실제 성과 순으로 정렬하여 즉시 제공

## ⚠️ 실패에서 배운 것들

| 문제 | 내용 |
|------|------|
| **엔티티 추출 오류** | 일부 "사람" 페이지가 실제로는 장비 이름 → 인간 검토 필수 |
| **공급업체 중복 문제** | 동일 하청업체가 10년간 3가지 ABN으로 등록 → 지속적인 정리 작업 필요 |

### 💡 가장 중요한 설계 결정
> **"AI 전에 폴더 구조부터"**
- 문서를 폴더 접두사로 먼저 분류 → LLM은 폴백(fallback)으로만 사용
- 이 방식으로 **AI 분류 호출의 95% 제거**
- 완벽한 추출보다 **중요한 문서의 80%** 확보가 더 효과적

---

## 🏆 결론: 위키가 해자(Moat)다

> *"모델은 더 나은 것으로 교체될 것이다. API 환경도 바뀔 것이다. 하지만 지식이 구조화되고, 연결되고, 복리로 쌓인다면 그것은 중요하지 않다. 위키가 해자(Moat)이지, 모델이 아니다."*

- 수십 년의 프로젝트 히스토리가 아무도 열지 않는 폴더에 잠겨 있다면 → **그것은 아카이브가 아니라 미개발 인텔리전스**
- 필요한 도구는 이미 존재하며, **책상 아래 단일 GPU**로 실행 가능

### 실천 방법
1. **지금 가진 것에서 시작하라**
2. **구조화하라**
3. **복리로 쌓이게 하라**

## 📌 Karpathy 개인 위키 vs TCD Group 비즈니스 위키 비교

| 항목 | Karpathy (개인) | TCD Group (비즈니스) |
|------|----------------|---------------------|
| 규모 | 단일 연구 주제 | 40년 역사, 1,800+ 프로젝트 |
| 레이어 | 3단계 | 4계층 |
| 데이터 | 연구 논문, 아티클 | 계약서, 안전 보고서, 견적서 등 |
| 실행 환경 | 클라우드 API | 완전 로컬 (데이터 보안) |
| 목적 | 개인 지식 합성 | 기관 지식 보존 및 비즈니스 인텔리전스 |
