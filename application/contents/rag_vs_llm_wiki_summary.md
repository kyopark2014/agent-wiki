# "RAG는 충분하지 않다 — Karpathy가 보여준 다음 단계"

> 원문: [LinkedIn Pulse - RAG Is Not Enough. Karpathy Just Showed Us What Comes Next.](https://www.linkedin.com/pulse/rag-enough-karpathy-just-showed-us-what-comes-next-ravindra-au7ac/)

---

## 🔑 핵심 배경

2026년 4월 3일, AI 연구자 **Andrej Karpathy**가 X(트위터)에 올린 포스트가 **1,600만 뷰**를 기록했어요.

> *"요즘 LLM을 활용해 개인 지식 베이스를 만드는 게 매우 유용하다. 코드 조작보다 지식 조작에 더 많은 토큰을 쓰고 있다."*

---

## 🏗️ Karpathy가 만든 것: LLM Wiki (지식 컴파일러)

3단계로 구성된 아주 단순한 아키텍처:

| 단계 | 내용 |
|------|------|
| **1. Raw** | 논문, GitHub, 웹 아티클, PDF 등을 `raw/` 폴더에 그냥 던져 넣기 (전처리 없음) |
| **2. Compile** | LLM이 문서를 읽고 → 요약, 개념 설명, 상호 참조가 담긴 **구조화된 위키(Markdown)** 자동 생성 |
| **3. Query** | 원본 문서가 아닌 **컴파일된 위키**에 질문 |

> 결과: 하나의 연구 주제에 대해 **약 100개 문서, 40만 단어** 분량의 위키가 자동 생성됨 (박사 논문보다 긴 분량!)

---

## ✅ LLM Wiki의 장점

1. **구조 위에서 추론** — RAG처럼 단편적인 청크가 아닌, LLM이 직접 만든 연결된 구조 위에서 답변
2. **검색 인덱스가 아닌 이해의 레이어** — "어디에 있는가"가 아닌 "무슨 의미인가"를 알려줌
3. **청킹(Chunking) 문제 탈출** — 512토큰 분할의 한계를 넘어 문서 전체를 이해하고 합성
4. **지식이 함께 진화** — 새 문서 추가 시 위키 전체가 업데이트되며 모순도 자동 감지

---

## ⚠️ 한계점 (솔직한 평가)

- 💸 **컴파일 비용** — 대규모 문서에는 비용이 많이 들 수 있음
- 📝 **프롬프트 품질 의존** — 컴파일 프롬프트가 나쁘면 위키 품질도 나빠짐
- 🚨 **환각(Hallucination) 전파 위험** — 컴파일 단계의 오류가 위키 전체에 퍼질 수 있음
- 👤 **현재는 개인 도구** — 팀 공유, 버전 관리 등 협업 인프라 미비

---

## ⚖️ LLM Wiki vs RAG — 언제 뭘 쓸까?

| **LLM Wiki가 유리한 경우** | **RAG가 유리한 경우** |
|---|---|
| 여러 문서를 넘나드는 복잡한 질문 | 실시간으로 변하는 대규모 데이터 |
| 깊은 이해와 합성이 필요할 때 | 단순 사실 조회 |
| 전문가가 직접 큐레이션한 코퍼스 | 출처(provenance) 추적이 중요할 때 |
| 구조적 추론이 필요한 질문 | 빠른 배포가 필요할 때 |

> 💡 **핵심 비유**: RAG는 데이터베이스 쿼리, LLM Wiki는 제2의 두뇌 — 경쟁 관계가 아니라 상호 보완 관계!

---

## 🔮 저자의 결론

> *"우리는 '검색의 시대'에서 '이해의 시대'로 넘어가고 있다."*

- 앞으로의 엔지니어링 과제는 "어떻게 올바른 청크를 찾을까?"가 아니라 **"어떻게 지식을 올바른 구조로 컴파일할까?"** 로 바뀔 것
- 잘 구조화된 LLM 지식 베이스를 구축하는 사람들이 **복리처럼 쌓이는 전문성 우위**를 갖게 될 것

---

## 📋 RAG → LLM Wiki 마이그레이션 로드맵

1. **Phase 1** — 현재 RAG에서 가장 잘 되는 것과 가장 안 되는 것 파악
2. **Phase 2** — 소규모 고품질 코퍼스(50~100개 문서) 큐레이션
3. **Phase 3** — 컴파일 프롬프트 정교하게 설계 (가장 중요한 단계!)
4. **Phase 4** — 첫 컴파일 실행 후 결과 비판적으로 검토
5. **Phase 5** — 동일 쿼리를 RAG vs LLM Wiki에 3주간 병렬 테스트
6. **Phase 6** — 증분 컴파일(Incremental Compilation) 구현
7. **Phase 7** — 데이터 기반으로 무엇을 교체하고 유지할지 결정

---

> 한마디로, **RAG가 "검색"이라면 LLM Wiki는 "이해"**.
> Karpathy가 보여준 이 접근법은 AI가 단순히 정보를 찾아주는 것을 넘어,
> 지식을 구조화하고 연결하는 새로운 패러다임을 제시하고 있다. 🚀
